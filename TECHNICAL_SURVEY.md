# Technical Survey: NL2SQL Context-Aware System

## 4.0 Software Requirements
This section outlines the essential software tools, libraries, and frameworks needed to implement and run the NL2SQL system effectively, covering the dual-stack architecture (Spring Boot/React) and the Python-based NLP engine.

### 4.0.1 Operating System
*   **Linux (Ubuntu 22.04 LTS Recommended):** Preferred for production environments and seamless compatibility with Python's `uv` package manager and spaCy models.
*   **Windows 10/11:** Supported via WSL2 (Windows Subsystem for Linux) for optimal Python performance.
*   **macOS:** Fully compatible for development purposes.

### 4.0.2 Programming Languages
*   **Python 3.12+:** The primary language for the NLP translation engine, chosen for its robust linguistic libraries (spaCy) and SQL manipulation tools (sqlglot).
*   **Java 17/21:** Used for the Spring Boot microservices backend to handle business logic, security, and orchestration.
*   **JavaScript/TypeScript (ES6+):** Utilized for the React-based frontend to provide a dynamic user interface.
*   **SQL (PostgreSQL Dialect):** The target language for all generated queries.

### 4.0.4 Environment Setup
A clean development environment is managed using the following methodologies:
*   **Python:** Uses `uv` for lightning-fast dependency resolution and virtual environment management.
    ```bash
    uv sync
    uv run uvicorn api.main:app --reload
    ```
*   **Java/Spring Boot:** Managed via Maven or Gradle, ensuring reproducible builds within the microservice architecture.
*   **React:** Managed via `npm` or `yarn`, utilizing Vite or Webpack for bundling.

### 4.0.5 Development Tools
*   **VS Code / IntelliJ IDEA:** Recommended IDEs with extensions for Python, Java, and React.
*   **Postman / Insomnia:** For testing the FastAPI and Spring Boot REST endpoints.
*   **DBeaver / pgAdmin:** For PostgreSQL database management and schema inspection.
*   **Git:** Essential for version control and collaborative development.

## 4.2 Hardware Requirements
The system is designed to be efficient but benefits from modern hardware due to the linguistic processing overhead of spaCy.

### 4.2.1 Minimum Requirements (Development/Testing)
*   **CPU:** Quad-core processor (e.g., Intel i5 or AMD Ryzen 5).
*   **RAM:** 8 GB (Minimum for running the API and a local PostgreSQL instance).
*   **Storage:** 5 GB free space for project files, models, and virtual environments.

### 4.2.2 Recommended Requirements (Production/Large Schemas)
*   **CPU:** 8-core processor or higher.
*   **RAM:** 16 GB or more (Critical when handling large `metadata.json` files and high-concurrency spaCy Parsing).
*   **Storage:** SSD with 20 GB+ free space for faster model loading and database I/O.
*   **GPU:** Not required (spaCy `en_core_web_sm` is optimized for CPU-based inference).

## 4.3 Network & Dataset Considerations
Unlike traditional ML models, this system is **context-driven** and derives its knowledge from the database schema itself.

### 4.3.1 Metadata-Driven "Dataset"
The "dataset" for this project is primarily the `metadata.json` file generated by the **Schema Inspector**. It contains:
*   **Table Descriptions:** Human-readable context for fuzzy matching.
*   **Column Data Types:** Decides whether to generate numeric comparisons or string matches.
*   **Distinct/Sample Values:** Acts as a training-free reference for resolving filter values (e.g., mapping "Mumbai" to the `city` column).

### 4.3.2 Query Resolution Logic
*   **Fuzzy Semantic Matching:** Uses `RapidFuzz` (WRatio) to map natural language tokens to schema elements without hardcoding.
*   **Deterministic Translation:** Leverages a rule-based generator (`SQLGenerator`) to ensure valid PostgreSQL syntax, avoiding the "hallucinations" common in LLM-based approaches.
*   **Semantic Evaluation:** Utilizes `sqlglot` for comparing predicted SQL against ground-truth queries during the performance evaluation phase.

### 4.3.3 Dataset Usage (Performance Evaluation)
The `evaluate_performance.py` module runs a suite of test cases (Basic, Filter, Aggregation, Ordering, Comparison, Date) to measure:
*   **Semantic Accuracy:** Whether the generated SQL logic matches the intended outcome.
*   **Robustness:** Handling variations in user phrasing.
*   **Latency:** Time taken for the spaCy-to-SQL pipeline to complete.
